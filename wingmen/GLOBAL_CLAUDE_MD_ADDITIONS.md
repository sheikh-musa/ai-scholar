## WINGMEN NERVOUS SYSTEM

This workspace is part of the Gazzabyte Nervous System. All repos under
`~/gazzabyte/` that are registered in `wingmen_brain_config` are monitored
by the `brain_sync` scheduled task every 4 hours.

### Architecture Summary

- **brain_sync** (every 4h): Scans all `brain_sync_enabled` repos. Detects
  contradictions. Writes operational snapshot to Supabase `wingmen_brain`.
  Also writes `~/gazzabyte/BRAIN.md` for local reference.
- **morning_brief** (6AM SGT): Reads latest snapshot. Sends Telegram brief.
- **session_compress** (2AM SGT): Detects context entropy. Checkpoints session.
- **memory_sync** (midnight SGT): Diffs snapshots. Flags identity-level changes
  that should update claude.ai memory edits.

### STATUS.md Protocol (MANDATORY FOR ALL ACTIVE REPOS)

Every repo with `status_category = 'active'` in `wingmen_brain_config` MUST
have a `STATUS.md` at its root. This is not optional — brain_sync cannot
generate an accurate snapshot without it.

**Update STATUS.md after every:**
- Completed build phase
- Deploy to Vercel or any environment
- Blocker encountered or resolved
- Client or revenue signal received
- Significant work session ends

**STATUS.md format — use this EXACTLY (field names must match):**

```markdown
# STATUS — [Product Display Name]

Last Updated: [ISO 8601 timestamp, e.g. 2026-03-31T08:00:00Z]
Phase: [description of current phase]
Status: [building|testing|deployed|blocked|done]
Deploy URL: [full URL or omit line if not deployed]
Health: [green|yellow|red]

## Completed
- [item]
- [item]

## Blocked
- [blocker description — what it is and why]

## Next Up
- [item]

## Revenue Signals
- [buying signal, client feedback, payment event]

## Questions for CTO (claude.ai)
- [strategic question for the planning brain]
```

**Health definitions:**
- `green`: actively being built or deployed and working
- `yellow`: stalled — no commits in 7+ days despite being active
- `red`: blocked or broken — needs attention

### Commit Attribution Policy

#### Open-Source / Waqf Repos

For repos where `is_open_source = true` in `wingmen_brain_config`
(currently: `hifz_companion`, `tadabbur`):

When writing commit messages and PR descriptions, you are operating on a
public repository. Apply these rules:

**NEVER include in commits:**
- "Claude Code" or any indication an AI authored the work
- Internal Gazzabyte codenames, project names, or references
- References to Wingmen, brain_sync, the orchestrator, or any internal tooling
- Internal Supabase table names or infrastructure details
- Co-Authored-By lines attributing authorship to AI

**DO include:**
- Conventional commit format: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`
- Clear, human-readable description of what changed and why
- GitHub issue references if applicable (e.g. `Closes #42`)

Write commits as a thoughtful human contributor would. The code is waqf —
it belongs to the ummah. Represent it professionally.

#### Private Repos

No special restrictions. Normal commit messages apply.

### Wingmen Job Queue Protocol

When Musa asks you to build, fix, or deploy something via Telegram:

1. Check `~/gazzabyte/BRAIN.md` or query `wingmen_brain` for current product state
2. Create a job in the `jobs` table with appropriate metadata
3. Execute the work
4. Update `STATUS.md` in the affected repo when done
5. Write a `build_runs` entry
6. Report back to Musa via Telegram with what was done

Do NOT silently drop work. Every task Musa requests must result in either
a completed job entry or an explanation of why it could not be completed.

### Context Freshness Rule

Before making any operational assertion about a product (e.g. "IhsanDMS was
deployed last Tuesday"), verify against:
1. `~/gazzabyte/BRAIN.md` freshness header (when was it last generated?)
2. The STATUS.md in the actual repo if BRAIN.md is > 4h old

Do NOT assert facts from memory alone. Verify against files or Supabase.
This is the "skeptical memory" rule — treat your own context as a hint,
not ground truth.

### Graceful Degradation

If brain_sync has not run recently and BRAIN.md is stale:
- Still respond to Musa's requests
- Surface the staleness: "BRAIN.md is [X]h old — I'm working from potentially stale state."
- Offer to run brain_sync manually if Musa wants fresh data

The system must never be unusable because one component is down.
